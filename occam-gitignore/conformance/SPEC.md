# occam-gitignore v1 Specification

Status: **Draft** (will be promoted to Stable on tag `v1.0.0`).

This document defines the algorithm that any conformant implementation
MUST implement. It is paired with the
[conformance suite](./README.md) which is the source of truth for byte
behavior. If the spec and the suite disagree, the suite wins.

## 1. Inputs

A conformant implementation accepts:

- **Templates**: a set of `<feature>.gitignore` files. The set is
  content-addressed; its `templates_version` is
  `sha256:<first 12 hex of sha256(name1 \0 content1 \0 name2 \0 content2 ...)>`
  over the files sorted by name.
- **Rules table**: a JSON object `{ "version": str, "rules": [...] }`.
  Each rule is `{ "features": [str], "patterns": [str] }`. The
  declared version, if it starts with `sha256:`, MUST equal the
  content-hash of the canonical encoding (sorted features and unique
  sorted patterns per entry; entries sorted by `(features, patterns)`;
  `json.dumps(..., sort_keys=True, ensure_ascii=False)`).
- **Tree**: an ordered tuple of POSIX path strings (input order is
  irrelevant to the output; see §3).
- **Options**:
  - `extras`: tuple of pattern strings appended verbatim
  - `include_comments`: whether section headers and provenance line
    are emitted
  - `include_provenance`: per-line `# <feature>` annotations

## 2. Fingerprint

A fingerprint is computed by 11 explicit detectors plus an implicit
`common` feature. A path matches a detector iff a fixed predicate
(see [`fingerprint.py`](../packages/occam-gitignore-core/src/occam_gitignore_core/fingerprint.py))
returns true.

For each detected feature, the witness path emitted is the
**lexicographically smallest** matching path. This makes the evidence
tuple invariant to input order.

If at least one detector matches, the implicit feature `common` is
added with witness `(implicit)`.

## 3. Determinism contract

For fixed `(fingerprint, options, templates_version, rules_table_version)`
the function `generate(...)` is **pure**:

- no I/O,
- no clock,
- no environment lookup,
- no randomness.

In particular: `fp(tree) == fp(shuffle(tree))` for both `features` and
`evidence`, and `generate(...).content` is byte-identical across calls
on byte-identical inputs.

## 4. Generation algorithm

```
features := sorted(fingerprint.features)
collected := []
for f in features:
    if templates.has(f): collected += templates.get(f)
collected += rules_table.extras_for(features)
collected += [Rule(p, USER_EXTRA) for p in options.extras]
ordered := stable_dedup(collected)
content := render(ordered, options, versions)
content_hash := "sha256:" + sha256(content).hex
provenance_hash := "sha256:" + sha256(
    core_version || \0 ||
    templates_version || \0 ||
    rules_table_version || \0 ||
    content_hash || \0
).hex
```

`stable_dedup` keeps the first occurrence per pattern, then sorts by
`(source_priority, pattern)` where `TEMPLATE < MINED < USER_EXTRA`.

`render` emits, when `include_comments` is true:

```
# Generated by occam-gitignore. Deterministic. Do not edit by hand.
# core=<core_version> templates=<templates_version> rules_table=<rules_table_version>
```

then sections separated by `# --- <source> ---` lines, then patterns,
then a single trailing newline. Encoding is UTF-8 without BOM. Line
endings are `\n` only (LF). The file MUST end with exactly one `\n`.

## 5. Hashes

- `content_hash`: full hex sha256 of the UTF-8 bytes of `content`.
- `provenance_hash`: as defined above. This commits to the full chain
  so a verifier can prove "this output came from these inputs" without
  publishing the inputs.

## 6. Versioning

This specification is `v1`. Any change that would alter the bytes of
an existing conformance case is a **major** version bump and the
affected cases MUST be renamed in the suite.

## 7. Reference implementation

The Python implementation in this repository is the reference. When in
doubt, the conformance suite is the arbiter; when the suite is silent,
the reference is the arbiter.
